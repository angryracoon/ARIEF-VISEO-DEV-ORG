## How card content reaches an agent

When Workboard **dispatches** a card to an agent (via dashboard, `openclaw workboard dispatch`, or `/workboard dispatch`), the Gateway:

1. **Claims the card** and moves it to `running`
2. **Builds a worker prompt** containing the card's title, bounded notes, context (attempts, comments, links, proof, artifacts, parent results), the assigned board, and the Workboard worker protocol
3. **Injects the claim token** so the agent can call heartbeat/complete/block tools
4. **Starts a subagent session** — the card content arrives as the opening prompt of that session

The agent doesn't need to "fetch" the card — the content is delivered directly when the session starts.

---

## Workboard agent tools available

Once Workboard is enabled, these tools are exposed to agents:

| Tool | Purpose |
|---|---|
| `workboard_list` | List cards with filters |
| `workboard_read` | Read full card + worker context |
| `workboard_create` | Create a new card |
| `workboard_link` | Link parent → child cards |
| `workboard_claim` | Claim a card (moves it to `running`) |
| `workboard_heartbeat` | Keep the claim alive during long runs |
| `workboard_release` | Release claim after completion/handoff |
| `workboard_complete` | Mark done with proof/artifacts |
| `workboard_block` | Mark blocked with reason |
| `workboard_comment` | Add handoff notes |
| `workboard_specify` | Clarify a triage/backlog card into `todo` |
| `workboard_decompose` | Fan a card into linked child cards |
| `workboard_runs` | View run-attempt history on a card |
| `workboard_proof` / `workboard_attachment_*` | Attach evidence/files |
| `workboard_notify_*` | Subscribe to and read card events |
